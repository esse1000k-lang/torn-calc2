# IP 3명 허용 + 30일 미로그인 자동 삭제 — 영향·주의사항

## 1. 과거 가입 규칙과의 관계 (꼬임 여부)

### 1.1 현재 로직이 꼬이지 않는 부분

| 항목 | 설명 |
|------|------|
| **signupIp** | 지금은 `/api/register`(리듬 가입)에서만 저장함. **과거에 지갑/다른 경로로 가입한 회원**은 `signupIp`가 없을 수 있음. `checkDuplicateIp`는 `u.signupIp === clientIp`인 사람만 센다. → 옛날 회원( signupIp 없음)은 “같은 IP 3명”에 **포함되지 않음**. 따라서 **기존 로직과 충돌 없음**. |
| **lastLoginAt** | 가입·로그인 시에만 넣고, **없는 회원은 삭제 대상에서 제외**함(`if (!u.lastLoginAt) return true` → 유지). 그래서 **과거에 가입한 회원(lastLoginAt 없음)**은 자동 삭제되지 않음. |
| **기존 회원** | `lastLoginAt` 없음 → 유지, `signupIp` 없음 → IP 카운트 안 됨. **과거 회원가입 규칙 때문에 현재 로직이 꼬이지는 않음**. |

### 1.2 의도와 다를 수 있는 부분 (선택 검토)

| 항목 | 내용 |
|------|------|
| **같은 IP “3명”의 범위** | “3명”에 포함되는 건 **signupIp가 저장된 회원**뿐임. 옛날에 가입해서 `signupIp`가 없는 회원은 같은 IP여도 카운트 안 됨. 그래서 **이론상** “같은 IP에서 옛날 2명 + 새로 3명”처럼 **한 IP에 5명**도 가능함. 완전히 “IP당 3명”만 허용하려면 옛날 데이터에 signupIp를 채우는 마이그레이션이 필요함. |
| **승인 대기(approved: false)** | 30일 미로그인 삭제 시 **승인 대기 회원도 제외하지 않음**. 다만 가입 시 `lastLoginAt`을 넣으므로, **승인 안 되고 30일 지나면** 삭제됨. “승인 대기는 삭제하지 않는다”로 바꾸고 싶으면 `u.approved === false`면 `return true`(유지) 추가하면 됨. |

## 2. 자동 삭제 시 생길 수 있는 문제점

| 문제 | 설명 | 대응 |
|------|------|------|
| **채팅/피드에 “삭제된 사용자”** | 채팅·피드 등은 `userId`만 저장하고, 표시할 때 `users`에서 조회함. 회원이 삭제되면 해당 유저를 찾지 못해 **이름/프로필이 비거나 “알 수 없음”**처럼 보일 수 있음. | 데이터는 그대로 두고, 표시만 “삭제된 사용자” 등으로 처리하는 방식이 일반적. 필요하면 클라이언트/API에서 `user` 없을 때 fallback 문구 넣으면 됨. |
| **referrer(추천인)** | 가입 시 `referrer`는 문자열 등으로 저장. **추천인 본인이 나중에 미활동 삭제**돼도, 가입자 쪽 `referrer` 필드는 그대로라서 **로직 꼬임은 없음**. 통계에서 “추천인 id로 유저 조회”만 한다면 삭제된 id는 조회 실패할 수 있음. | referrer를 id가 아니라 닉네임 등으로만 쓰고 있다면 문제 없음. id로 조회한다면 null 처리 등만 하면 됨. |
| **동시 쓰기** | `deleteInactiveUsers`가 `readUsers` → 필터 → `writeUsers` 한 번에 하므로, 그 사이에 다른 API가 `writeUsers`를 호출하면 **한쪽이 덮어쓸 수 있음**. | 단일 서버·트래픽이 크지 않으면 실무에서 흔히 감수하는 수준. 필요하면 db 쪽에 lock/버전 등 추가 가능. |

---

## 3. 요약

- **과거 회원가입 규칙 때문에 현재 로직이 꼬이는 부분은 없음.**  
  - 옛날 회원: `signupIp` 없음 → IP 제한에 안 걸림.  
  - `lastLoginAt` 없음 → 미활동 삭제 대상 아님.
- **선택 검토:**  
  - “진짜 IP당 3명”만 허용하려면 예전 회원에 대한 `signupIp` 보정이 필요.  
  - 승인 대기 회원은 미활동 삭제에서 빼고 싶으면 `approved === false`일 때 유지하도록 조건 추가하면 됨.
- **자동 삭제로 인한 부작용**은 주로 **채팅/피드에서 삭제된 userId 표시** 정도이고, referrer·가입 로직 자체가 꼬이는 건 아님.
