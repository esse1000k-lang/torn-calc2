# 리듬 패턴 잠금(Rhythm-Tap Login) 검토 — 비밀번호 대체

## 1. 요약 결론

- **완전 대체(비밀번호 없이 리듬만)**: 권장하지 않음. 엔트로피·재현성·표준 대응이 부족함.
- **보조 인증(비밀번호 + 리듬)** 또는 **편의 옵션(리듬으로 빠른 잠금 해제)** 로 도입하는 것은 검토 가능.

---

## 2. 작동 원리 정리

| 단계 | 내용 |
|------|------|
| **설정** | 사용자가 박자 패턴 입력 (예: 쿵-쿵-팍 → 구간 1초, 0.5초, 0.2초) |
| **로그인** | 동일 영역(로고/버튼)을 같은 박자로 탭 → 검증 |
| **보정** | 구간 간 **상대 비율**로 비교, ±15% 등 오차 허용 |

---

## 3. 장점

- **기억 부담 감소**: 숫자/문자 암기 대신 “리듬”만 기억.
- **피싱/키로거에 강함**: 타이밍 데이터는 전통적인 비밀번호 탈취 경로와 다름.
- **접근성**: 시각·청각 보조와 결합 가능(진동/소리 피드백).
- **차별화**: 커뮤니티 앱의 재미·이미지에 도움.

---

## 4. 한계 및 리스크 (비밀번호 대체 시)

### 4.1 엔트로피(조합 수)

- 비밀번호: 글자/길이에 따라 수십~수백 비트.
- 리듬: 구간 개수 N, 각 구간을 몇 단계로 양자화하느냐에 따라 제한적.
  - 예: 5구간, 각 10단계 → 10^5 = 10만 가지. 8자 영숫자보다 훨씬 적음.
- **짧은 패턴(3~4박)** 이면 브루트포스/사전식 시도에 취약.

### 4.2 재현성(일관성)

- 기기/브라우저/부하에 따른 **타이머 지터**, 터치/클릭 **지연** 차이.
- 사용자 컨디션(급할 때, 피곤할 때)에 따라 입력 궤적이 달라짐.
- ±15%처럼 넓히면 보안이 약해지고, 좁히면 “본인인데도 실패”가 늘어남.

### 4.3 저장·검증 방식

- **비밀번호**: bcrypt 등으로 해시 저장, 상수 시간 비교로 타이밍 공격 완화 가능.
- **리듬**: 구간 길이 배열을 그대로 저장하면 유출 시 그대로 복제 가능.  
  → “리듬 비밀”을 해시에 넣어서 검증하거나, 서버는 **비율/구간 수**만 검증하고 세부 값은 클라이언트에서만 쓰는 식의 설계가 필요.

### 4.4 표준·정책

- “비밀번호 대체”로만 쓰면, 감사·규정에서 “비밀번호 정책 미준수”로 보일 수 있음.
- 2FA·복구 절차 등 기존 보안 표준과의 정합성도 고려 필요.

**안 지키면 어떻게 되나?**

“비밀번호 정책 미준수”는 **누가 요구하느냐·어디에 쓰이느냐**에 따라 결과가 달라진다.

| 상황 | 누가 / 어떤 기준을 봄 | 안 지켰을 때 가능한 결과 |
|------|------------------------|---------------------------|
| **법·규제** | 금융감독원, 개인정보보호법, PCI-DSS(결제), 의료·공공 등 **법령·감독기관** | 해당 법이 “비밀번호 또는 이에 상응하는 인증”을 요구하면, **리듬/패턴만**으로는 “상응” 인정을 못 받을 수 있음. → **시정 요구, 과태료, 인가 취소** 등 (업종·규정에 따라 다름). |
| **계약·인증** | 거래처, 대기업, **ISO 27001·SOC2** 등 인증/감사 | 계약서·인증 기준에 “비밀번호 정책 준수”가 있으면, **미준수 시 인증 불통과·계약 위반**으로 이어질 수 있음. 재계약/입찰에서 불리. |
| **플랫폼·스토어** | 앱스토어, Google, 결제사(PG), OAuth 제공자 등 | 정책에 “비밀번호 또는 강한 인증”을 요구하면, **리듬만 쓰는 앱**이 심사에서 걸리거나 가이드 위반으로 제재받을 수 있음. |
| **자체 서비스(커뮤니티 등)** | 외부 감사·규정 없음. 자체 정책만 있음. | **법적/계약적 제재는 없음.** 다만 나중에 **규정 도입·투자·매각·협력** 시 “당시 비밀번호 정책 미준수”가 이슈로 떠오를 수는 있음. |

**정리**

- **금융·결제·의료·공공**처럼 규제가 있는 서비스: “비밀번호 대체만” 쓰면 **규정 위반·시정·과태료** 가능성이 있으니, 기준을 먼저 확인하고 비밀번호를 유지하거나 “상응 인증”으로 인정받을 수 있는 방식(예: FIDO, 비밀번호+2FA)을 쓰는 편이 안전하다.
- **TornFi처럼 커뮤니티·자체 서비스**이고, 외부 감사·규제·계약상 “비밀번호 정책”이 **강제되지 않는** 환경이면: **안 지켜도 당장 벌금이나 인증 취소 같은 일은 없다.** 다만 “표준적으로는 비밀번호를 쓰는 게 맞다”는 권고 수준이고, 나중에 규정·감사가 생기면 그때 맞추면 된다.
- **위험을 줄이고 싶으면**: 비밀번호는 두고, 리듬/패턴/이미지 탭을 **2FA나 잠금 해제**로만 쓰면 “비밀번호 정책”과 충돌하지 않는다.

### 4.5 스누핑·모방

- 숨어서 **손동작/화면 탭**을 관찰당하면 리듬이 그대로 노출됨.
- 짧고 단순한 패턴이면 몇 번 보여주면 재현 가능.

---

## 5. 권장 방향

### A. 비밀번호 완전 대체 — 비권장

- 엔트로피·재현성·표준 측면에서 단일 인증 수단으로 쓰기엔 무리가 있음.

### B. 보조 인증(2FA) — 권장

- **1차**: 기존 닉네임 + 비밀번호  
- **2차**: 리듬 탭 (선택)
- 비밀번호 유출만으로는 로그인 불가, 보안 강화 + UX 차별화 동시에 가능.

### C. “빠른 잠금 해제” 전용 — 권장

- 앱/탭을 잠금해 두었다가, **같은 기기에서** 리듬만으로 잠금 해제.
- 실제 “로그인”(서버 인증)은 비밀번호(또는 세션)로만 수행.
- 기기 분실 시에는 비밀번호로만 재로그인하도록 하면, 리듬은 “편의 기능”으로 한정됨.

### D. 비밀번호 + 리듬 “둘 중 하나” — 조건부

- “비밀번호 **또는** 리듬으로 로그인”은 공격면이 넓어짐(둘 중 약한 쪽이 목표).
- 리듬을 **반드시** 더 강한 정책(구간 수·복잡도)으로 제한하고, 로그 시도 제한·모니터링을 엄격히 할 때만 검토.

---

## 6. 구현 시 기술 포인트 (참고)

- **저장**: 구간 간 **상대 비율** 배열만 저장 (절대 시간 X).  
  예: `[1.0, 0.5, 0.2]` → 정규화된 비율로 저장.
- **검증**:  
  - 입력 구간 비율 벡터와 저장된 비율 벡터의 코사인 유사도, 또는 구간별 비율 오차가 전부 ±15% 이내인지 확인.
- **엔트로피 확보**:  
  - 최소 구간 수(예: 5~7), “너무 단순한” 패턴(전 구간 동일 등) 거부.
- **클라이언트**:  
  - `mousedown`/`mouseup`, `keydown`(스페이스), `touchstart`/`touchend`의 타임스탬프로 구간 길이 계산 후, 정규화해서 서버로 전송.
- **보안**:  
  - 리듬 데이터는 서버에서 bcrypt 등으로 해시해 저장하거나, 검증 시 상수 시간 비교로 타이밍 공격을 완화.

---

## 7. 최종 정리

| 용도 | 비밀번호 대체(단독) | 보조 인증(2FA) | 잠금 해제 전용 |
|------|---------------------|----------------|----------------|
| 보안 적합성 | 낮음 | 높음 | 중간(기기 한정) |
| UX/차별화 | 높음 | 높음 | 높음 |
| 구현·운영 부담 | 중간 | 중간 | 상대적으로 낮음 |

- **비밀번호를 없애고 리듬만 쓰는 방식은 권장하지 않고**,  
- **비밀번호는 유지한 채 “리듬 탭”을 2FA 또는 기기 잠금 해제용으로 추가**하는 구성을 추천합니다.

# 스마트폰 패턴 잠금(Pattern Lock) 검토 — 비밀번호 대체

## 1. 요약 결론

- **완전 대체**: 리듬보다는 낫지만, 강한 비밀번호 수준에는 미치지 못함. **저위험 서비스**에서만 단독 사용 검토.
- **보조 인증·기기 잠금 해제**: 리듬과 마찬가지로 **추천**. 저장·검증이 비밀번호와 동일하게 해시로 처리 가능해 구현이 단순함.

---

## 2. 작동 원리

- **3×3(또는 4×4) 그리드**의 점을 순서대로 연결해 경로를 그림.
- **노드 번호 순서**가 비밀(예: `1-2-5-8-7-4`). 중간에 있는 점은 건너뛸 수 없고, 이미 지난 점만 건너뛰기 가능(Android 규칙과 동일).

---

## 3. 엔트로피(조합 수)

| 방식 | 대략적인 경우의 수 | 비고 |
|------|---------------------|------|
| 4자리 PIN | 10^4 = 1만 | ~13비트 |
| 3×3 패턴 (길이 4~9) | 약 **39만** (연구 기준) | ~18~19비트 |
| 8자 영문+숫자 비밀번호 | 36^8 이상 | 40비트 이상 |

- 패턴은 **리듬 탭보다 조합 수가 훨씬 많고**, 4~6자리 PIN과 비슷하거나 조금 나은 수준.
- **최소 길이**(예: 5~6점 이상)를 강제하면 엔트로피가 올라감.

---

## 4. 장점

- **저장·검증이 비밀번호와 동일**: 노드 순서 문자열(예: `"125874"`)을 bcrypt 등으로 해시해 저장하면 됨. 기존 `passwordHash` 파이프라인 재사용 가능.
- **사용자 친숙함**: Android 잠금 화면 경험으로 이미 익숙함.
- **키로거/피싱에 강함**: 키보드 입력이 아니라 제스처라, 일반 비밀번호 탈취 경로와 다름.
- **재현성**: 리듬보다 안정적. 같은 경로만 그리면 되고, 타이밍/비율 오차가 없음.

---

## 5. 한계 및 리스크

### 5.1 스머지(흔적) 공격

- 화면에 남은 **기름/지문**으로 패턴 경로가 드러날 수 있음.
- 완전 대체 시 기기 물리 접근이 있으면 위험. 웹에서는 같은 PC에서만 쓰는 경우 상대적으로 덜함.

### 5.2 숄더 서핑

- **손가락 궤적**이 눈에 잘 들어와서, 옆에서 한두 번 보면 따라 그리기 쉬움. PIN보다 시각적 노출이 큼.

### 5.3 사용자 습관

- 연구에서 사람들이 **코너에서 시작**, **단순한 L·Z 형태**를 많이 씀. 실제 엔트로피는 이론값보다 낮을 수 있음.
- 최소 길이·금지 패턴(너무 단순한 형태) 정책으로 보완 가능.

### 5.4 표준·정책

- “비밀번호 대체”로만 두면 규정/감사에서 여전히 이슈 가능. 2FA나 보조 수단으로 쓰는 편이 정합성 좋음.

---

## 6. 리듬 vs 패턴 vs 비밀번호 (요약)

| 항목 | 리듬 탭 | 스마트폰 패턴 | 비밀번호 |
|------|---------|----------------|----------|
| 엔트로피 | 낮음 | 중간(PIN~단순 비밀번호 수준) | 높음(정책에 따라) |
| 재현성 | 타이밍 오차로 어려움 | 좋음 | 좋음 |
| 저장/검증 | 비율·해시 설계 필요 | **문자열 → 해시 (기존 방식과 동일)** | 해시 표준 |
| 스누핑/스머지 | 손동작 노출 | 궤적·흔적 노출 | 키 입력(키로거) |
| 사용자 친숙도 | 낮음 | **높음(Android 경험)** | 높음 |

---

## 7. 스마트폰 패턴 권장 용도

- **단독 비밀번호 대체**: 커뮤니티·저위험 서비스에서만 검토. 최소 5~6점 이상 강제 + 시도 횟수 제한 필수.
- **2FA(보조 인증)**: 비밀번호 후 패턴 한 번 더 → **추천**. 구현도 기존 비밀번호 해시 로직과 통일 가능.
- **기기/탭 잠금 해제**: 이미 로그인된 세션만 패턴으로 잠금/해제 → **추천**. 서버 인증은 비밀번호 유지.

정리하면, **스마트폰 패턴은 리듬보다 비밀번호 대체에 가깝고 구현도 쉽지만**, 보안·표준을 고려하면 **비밀번호를 유지하고 패턴은 2FA 또는 잠금 해제용**으로 쓰는 구성을 추천합니다.

# 그밖에 추천: 닉네임 + α (괜찮고 신박하고 보안 좋은 것들)

닉네임 + 리듬, 닉네임 + 패턴 말고도 아래처럼 **기억하기 쉽고, 구현 가능하고, 보안도 괜찮은** 대안들을 정리했다. 2FA·잠금 해제·저위험 서비스 단독 인증 등 용도에 맞게 골라 쓰면 된다.

---

## 1. 이미지 + 탭 포인트 (Picture Pass-Points) ⭐ 추천

**방식**: 사용자가 **한 장의 이미지**를 선택(또는 업로드)한 뒤, 그 이미지 위에서 **순서대로 3~5곳을 클릭/터치**. 로그인 시 같은 이미지를 보여주고 같은 위치를 같은 순서로 누르면 통과.

| 장점 | 한계 |
|------|------|
| **이미지 자체가 비밀의 일부** → DB만 털려도 이미지+위치를 모르면 재현 불가 | 위치를 픽셀 단위로 저장하면 해상도/화면 크기마다 달라짐 → **그리드로 양자화**(예: 10×10) 후 “칸 번호 순서”로 저장하면 안정적 |
| 기억하기 쉬움 (“고양이 눈, 코, 꼬리”) | 이미지 선택이 좁으면 예측 가능 → **이미지 풀을 넓히거나 사용자 업로드** 권장 |
| 키로거/피싱과 무관 | 스크린 캡처+클릭 위치 탈취 시에는 노출 가능 |

**저장**: 클릭한 **그리드 칸 번호 순서**를 문자열로(예: `"23,45,12,78"`) → bcrypt 해시. 기존 비밀번호 파이프라인 재사용 가능.

**용도**: 2FA, 또는 저위험 서비스에서 **닉네임 + 이미지 탭** 단독 로그인.

### 아이디 + 이미지 탭 포인트 — 해킹 가능성(공격 벡터)

| 공격 유형 | 가능성 / 설명 | 완화 방법 |
|-----------|----------------|-----------|
| **DB 유출만** | **어렵다**. 서버에는 탭 순서의 **해시**만 저장하고, **이미지는 저장하지 않거나** 사용자별로 암호화·분리 저장. 공격자는 “어떤 이미지에서 어떤 순서로 탭했는지”를 알 수 없음. | 탭 순서만 해시 저장. 이미지 ID/경로는 별도·최소 저장. |
| **브루트포스** | **이미지를 알아야** 시도 가능. 이미지를 안다고 가정하면: 그리드 10×10에서 5점 순서 = 100×99×98×97×96 ≈ 90억. 시도 제한(예: 5회 실패 시 잠금)이 있으면 현실적으로 불가에 가깝다. | **최소 탭 수 4~5**, **그리드 10×10 이상**, **시도 제한(5회 등) + 계정 잠금/캡차**. |
| **피싱 / 가짜 로그인** | **가능하다**. 공격자가 비슷한 이미지나 같은 이미지를 쓰는 가짜 로그인 페이지를 만들고, 사용자가 입력한 **클릭 좌표(또는 그리드 칸)** 를 그대로 수집할 수 있음. | 도메인·SSL 확인, 사용자 교육. 2FA로 비밀번호/OTP를 함께 쓰면 피싱만으로는 부족. |
| **스크린 캡처 + 클릭 로깅** | **가능하다**. 악성 앱/ malware가 화면과 클릭 위치를 함께 캡처하면 “이미지 + 탭 순서”가 그대로 유출됨. | 기기 보안(앱 신뢰, OS 최신화). 고위험 계정은 2FA 필수. |
| **숄더 서핑** | **가능하다**. 옆에서 이미지와 손가락으로 누르는 위치를 보면 재현 가능. | 공용 장소에서는 비밀번호나 2FA 보조 사용. |
| **사용자 선택 편향** | **실제 엔트로피 감소**. 사람들은 눈·코·중심·코너를 많이 탭함. 연구(PassPoints 등)에서 **예측 가능한 위치**가 많아 이론보다 조합 수가 줄어듦. | **최소 탭 수 늘리기**(5~6), **그리드 촘촘히**(예: 15×15), 또는 “특정 영역만 허용”이 아니라 **전 영역 동일 가중치**로 안내. |
| **이미지 풀 추측** | **이미지 개수가 적으면** 공격자가 “자주 쓰는 이미지 + 흔한 탭 위치” 조합을 우선 시도할 수 있음. | **이미지 풀을 크게**(수십~수백 장) 하거나 **사용자 업로드**로 이미지 자체를 비밀에 가깝게. |
| **리플레이** | 로그인 요청 시 **클릭 좌표를 그대로** 보내면, 그 패킷을 가로채서 재전송하면 될 수 있음. | **HTTPS 필수**, **세션/토큰**은 서버가 한 번만 허용(재사용 불가). 매 로그인마다 nonce나 챌린지 넣어 재전송 무효화. |
| **아이디 거대 목록 + 자동화** | 아이디 목록을 털어서 “아이디 + 흔한 이미지 + 흔한 탭 패턴”으로 대량 시도. | **시도 제한을 IP·계정 단위로**, **캡차/속도 제한**, **이미지가 비밀에 포함**되면 아이디만 알아도 이미지 모름 → 시도 난이도 상승. |

**요약**

- **DB만 유출**: 탭 순서 해시만 저장하고 이미지는 노출하지 않으면, **아이디 + 이미지 탭만으로는 재현이 사실상 어렵다**.
- **가장 현실적인 위험**: **피싱**(가짜 로그인 페이지에서 클릭 수집), **기기 감염**(화면+클릭 캡처), **숄더 서핑**.  
  → HTTPS, 시도 제한, 가능하면 **2FA(비밀번호 또는 OTP) 병행**으로 완화.
- **엔트로피**: 그리드 크기·탭 수·시도 제한을 넉넉히 주고, 사용자에게 “눈/코만 고르지 말고 다양한 위치”를 권장하면 브루트포스는 현실적으로 막을 수 있음.

## 2. 이모지/아이콘 시퀀스 (Emoji Lock)

**방식**: 4~6개 **이모지(또는 아이콘)** 를 **순서대로** 선택. 로그인 시 같은 그리드에서 같은 순서로 고르면 통과.

| 장점 | 한계 |
|------|------|
| **직관적·재미있고** 커뮤니티 앱 톤에 잘 맞음 | 그리드가 작으면 조합 수 감소 → **최소 12~20개 중 4~5개 선택** 등으로 엔트로피 확보 |
| “🐱→🌙→☕→🔑”처럼 **스토리로 기억** 가능 | 숄더 서핑으로 순서가 노출될 수 있음 |

**저장**: 선택한 이모지 코드 또는 인덱스 순서 문자열 → 해시. 패턴 잠금과 동일한 방식.

**용도**: 2FA, 기기 잠금 해제, 또는 닉네임 + 이모지 시퀀스 단독(저위험).

---

## 3. 방향 스와이프 시퀀스 (↑↓←→)

**방식**: **상·하·좌·우** 네 방향만 사용. 예: “위-위-왼쪽-아래-오른쪽”. 로그인 시 같은 방향 순서로 스와이프(또는 버튼 4개로 입력).

| 장점 | 한계 |
|------|------|
| **구현 단순**, 설명하기 쉬움 | 4방향 5칸 = 4^5 = 1024가지 → **길이를 6~7 이상**으로 하거나, 2FA용으로 쓰는 편이 안전 |
| 패턴처럼 “중간 점” 규칙이 없어 **입력이 단순** | |

**저장**: `"UUDLR"` 같은 문자열 → 해시.

**용도**: 2FA, 잠금 해제. 단독은 짧으면 엔트로피 부족하므로 **길이 강제 + 시도 제한** 필요.

---

## 4. 패스프레이즈 (문장 비밀번호)

**방식**: **긴 문장**을 비밀번호로 사용. 예: “우리집고양이는아침에만야옹해”. 단어 단위로 쪼개서 기억하기 쉽게 할 수도 있음(예: correct horse battery staple 스타일).

| 장점 | 한계 |
|------|------|
| **기억하기 쉽고 엔트로피 크다** (문장 길이만 해도) | 한글/영문 입력 환경, 자동완성 등 구현 시 주의 |
| 이미 많은 보안 가이드에서 **“비밀번호 대체”로 권장** | 짧거나 흔한 문장이면 사전 공격에 취약 → **최소 길이·금지 목록** 권장 |

**저장**: 기존 비밀번호와 동일하게 bcrypt. “닉네임 + 패스프레이즈”는 **가장 표준에 가깝고 보안도 좋은** 조합.

**용도**: **비밀번호를 완전 대체**하고 싶을 때 가장 무난한 선택. “닉네임 + 리듬/패턴”보다 감사·규정 대응이 수월함.

---

## 5. 타이핑 리듬 (Keystroke Dynamics) — 보조용

**방식**: **비밀번호는 그대로** 두고, “그 비밀번호를 **어떤 속도/간격으로 치는지**”를 등록. 로그인 시 비밀번호가 맞을 뿐 아니라, 키 누름 간 **타이밍 패턴**이 비슷할 때만 통과.

| 장점 | 한계 |
|------|------|
| **키로거가 비밀번호만 털어도** 타이밍이 맞지 않으면 통과 어려움 | 기기·키보드·피로도에 따라 타이밍이 달라져 **오차 허용** 설계 필요(리듬 탭과 유사) |
| 사용자는 “그냥 비밀번호만 치면 됨”이라 **UX 부담 적음** | 구현·튜닝 난이도가 있음 |

**용도**: **보조 인증**으로만. 비밀번호 + 타이핑 리듬 2단계.

---

## 6. 숨은 탭 횟수 (Tap Count Secret)

**방식**: 로고(또는 버튼)를 **몇 번씩 탭할지**만 비밀. 예: “3번 탭 → 1초 대기 → 1번 탭 → 1초 대기 → 4번 탭”. 리듬(구간 간 시간)과 **조합**하면 “몇 번 + 어떤 박자”가 되어 엔트로피가 늘어남.

| 장점 | 한계 |
|------|------|
| **리듬만 쓰는 것보다 조합 수 증가** (횟수 × 구간 수) | 횟수만 쓰면 3-1-4 같은 짧은 조합은 **브루트포스 가능** → 최소 길이·시도 제한 필요 |
| 구현이 리듬 확장으로 자연스러움 | |

**용도**: 리듬과 함께 “닉네임 + (탭 횟수 + 리듬)” 또는 2FA.

---

## 7. 한 번에 고르기 (Grid One-Time Choice)

**방식**: N×N 그리드에서 **로그인할 때마다** “이번에는 3번째 칸을 눌러라”처럼 **서버가 랜덤 위치를 정해** OTP처럼 한 번만 유효하게 줌. 사용자는 이메일/앱 알림으로 “오늘의 위치: (2,3)”을 보고 로그인 화면의 해당 칸을 누름.

| 장점 | 한계 |
|------|------|
| **매번 다른 값**이라 재사용·스누핑에 강함 | 이메일/앱 전달 채널이 안전해야 함. 구현은 OTP와 비슷 |
| 키로거·패턴 유출과 무관 | |

**용도**: 2FA(이메일/앱 OTP 대신 “그리드 한 칸”으로 할 수 있음). 단독 로그인보다는 보조용.

---

## 요약 표: 닉네임 + α 후보

| 방식 | 신박함 | 보안 | 구현 난이도 | 비밀번호 대체 단독 | 2FA/잠금해제 |
|------|--------|------|-------------|---------------------|----------------|
| **이미지 + 탭 포인트** | ⭐⭐⭐ | 높음 | 중간 | 가능(저위험) | ✅ 추천 |
| **이모지 시퀀스** | ⭐⭐⭐ | 중간 | 쉬움 | 가능(저위험) | ✅ 추천 |
| **방향 스와이프** | ⭐⭐ | 중간 | 쉬움 | 길이 길게 하면 가능 | ✅ 추천 |
| **패스프레이즈** | ⭐ | 높음 | 쉬움 | ✅ 가장 무난 | — |
| **타이핑 리듬** | ⭐⭐ | 보조로 유효 | 어려움 | ❌ | 보조만 |
| **탭 횟수 (+ 리듬)** | ⭐⭐ | 중간 | 쉬움 | 리듬과 조합 시 가능 | ✅ |
| **그리드 OTP** | ⭐⭐ | 높음 | 중간 | ❌ | ✅ |

- **완전 대체**하고 싶으면: **패스프레이즈**가 가장 안전하고 표준에 가깝고, **이미지 탭**은 신박하고 보안도 괜찮음.
- **닉네임 + 리듬/패턴 느낌**을 유지하면서 차별화하려면: **이모지 시퀀스**, **이미지 + 탭**, **탭 횟수 + 리듬** 조합을 추천.
