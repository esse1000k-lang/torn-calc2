# 리듬 패턴 로그인 — 컷트라인(판정) 완화 리포트

## 1. 현재 웹에서 빡센 이유

### 1.1 이미 적용된 방식 (좋은 점)

- **상대 비율로 판정**: 절대 시간(ms)이 아니라 **구간 간격의 비율**로 비교합니다.  
  - 전체 입력 시간 중 각 간격이 차지하는 비율을 구한 뒤, 그 비율만 비교합니다.  
  - 따라서 “전체적으로 빠르게/느리게” 두드려도 비율이 같으면 통과할 수 있는 구조입니다.
- **정규화(Normalization)**: 구간 배열을 합이 1이 되도록 정규화한 뒤, 이 비율 배열을 사용합니다.
- **양자화(Quantization)**: 비율을 일정 단위로 끊어서, 미세한 손떨림·네트워크 지연으로 인한 오차를 흡수합니다.
- **최소 구간 35ms**: 한 구간이 35ms 미만이면 오타(더블탭)로 보고 제거해, 비정상 입력을 걸러냅니다.

### 1.2 빡세게 느껴지는 핵심 원인

| 원인 | 설명 |
|------|------|
| **양자화 단위(step)가 작음** | 기존 `RHYTHM_QUANTIZE_STEP = 0.15` 사용. 비율이 0.15 단위로만 반올림되어, **실질 허용 범위가 약 ±7.5%**(step의 절반) 수준입니다. 한 구간만 경계를 살짝 넘어도 다른 버킷으로 떨어져 **키가 바뀌어 실패**합니다. |
| **전부 일치만 통과** | DTW 같은 “유사도 점수”가 아니라 **양자화된 키 문자열이 100% 일치해야만** 통과합니다. 한 구간만 달라도 실패합니다. |
| **긴장/속도 변화** | 사람은 매번 똑같은 속도로 두드리기 어렵습니다. 비율은 쓰지만, 양자화 버킷이 좁으면 “조금만 밀려도” 다른 버킷으로 가서 체감 난이도가 올라갑니다. |

즉, **“비율 + 양자화”는 이미 잘 쓰고 있으나, 양자화 단위(0.15)가 작아서 컷트라인이 빡세게 느껴진 것**에 가깝습니다.

## 2. 조사 자료 대비 현재 구현

| 조사 제안 | 현재 구현 상태 | 비고 |
|-----------|----------------|------|
| 1. 절대 시간 대신 **상대 비율**로 판정 | ✅ 적용됨 | `normalize(intervals)` → 비율 배열 사용 |
| 2. **DTW**로 유사도 측정 | ❌ 미적용 | 전부 일치 방식. 도입 시 라이브러리·검증 로직 확장 필요 |
| 3. **정규화 + 양자화** | ✅ 적용됨 | 정규화 후 `quantize(ratios)` 적용 |
| 4. **UX 보정** (시각/소리 피드백, 메트로놈 등) | ⚠️ 부분 적용 | 탭 시 파동·소리 등 시각 피드백 있음. 메트로놈·연속 학습은 미적용 |
| **추천: ±15%보다 완화** | ⚠️ 기존 step 0.15 ≈ ±7.5% 수준 | step 확대로 완화 가능 |

## 3. 적용한 개선 (보안 유지 + 쉬운 로그인)

### 3.1 양자화 단위 완화

- **변경**: `RHYTHM_QUANTIZE_STEP` **0.15 → 0.2**
- **효과**  
  - 한 비율이 “같은 버킷”으로 인정되는 범위가 넓어져, **타이밍 오차를 더 허용**합니다.  
  - 0.15 → 0.2로 키 버킷이 넓어지므로, **같은 리듬이라도 조금 달라도 같은 키로 묶일 확률**이 커집니다.
- **호환성**  
  - **기존 가입자**: 서버 검증 시 **0.2와 0.15 둘 다 시도**해, 예전에 0.15로 저장된 해시도 그대로 로그인 가능합니다.  
  - **신규 가입자**: 0.2 기준으로 해시 저장 → 처음부터 완화된 판정으로 등록·로그인합니다.

### 3.2 보안 유지

- **저장**: 여전히 **비율을 양자화한 키 문자열을 bcrypt로 해시**해 저장합니다. 원본 리듬/비율이 노출되지 않습니다.
- **판정**: “유사도 80% 통과”가 아니라 **같은 양자화 키 → 같은 해시**만 통과하므로, 예측 가능한 공격 면에서는 기존과 동일합니다.
- **완화 범위**: step만 0.2로 키는 조금 더 넓은 버킷으로 묶일 뿐, 여전히 **리듬 패턴이 맞아야** 같은 키가 나옵니다. 무작위 탭으로 통과하기는 어렵습니다.

---

## 4. 요약

- **빡세게 느껴진 이유**: 상대 비율 + 양자화는 이미 쓰고 있으나, **양자화 단위(0.15)가 작아** 작은 타이밍 차이에도 키가 바뀌어 실패하기 쉬웠음.
- **적용한 개선**: **step 0.2로 완화**하고, **기존 0.15 해시와의 호환**을 위해 로그인 시 두 step 모두 검사하도록 변경.
- **보안**: 저장·검증 방식은 그대로이며, “같은 키 = 같은 해시”만 통과하므로 보안을 유지하면서 **체감 난이도만 완화**된 상태입니다.

추가로 DTW, 메트로놈, 연속 학습 등을 도입하면 더 완화할 수 있으나, 현재 단계에서는 **양자화 단위 완화**만으로도 “컷트라인 빡세다”는 문제를 크게 줄일 수 있습니다.
