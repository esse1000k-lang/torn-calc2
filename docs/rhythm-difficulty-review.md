# 리듬 난이도 검토 — 탭 횟수 버그 수정 후

## 1. 결론

**3번에서 계속 실패했던 주된 원인은 “난이도(양자화 step)”가 아니라 “2번과 3번에서 탭 횟수 불일치”였다.**  
그 부분을 **같은 개수 구간만 잘라서 비교**하도록 수정했으므로, **난이도를 다시 올려도 된다.**

---

## 2. 지금까지 한 “완화” 정리

| 조치 | 내용 | 실제 원인과의 관계 |
|------|------|---------------------|
| **양자화 step 0.15 → 0.2** | 타이밍 오차 허용 범위 확대 | 3번 실패의 **진짜 원인 아님**. 탭 횟수만 맞아도 0.15로 통과 가능했을 수 있음. |
| **2·3단계 비교 시 “앞 N개 구간만 비교”** | 2번 4탭 / 3번 5탭처럼 횟수가 달라도 같은 리듬이면 통과 | **진짜 원인 해결**. 구간 개수 불일치로 키 길이가 달라져서 100% 실패하던 문제 제거. |

즉, **난이도 완화(step 0.2)는 “원인 오해”에 따른 대응**이었고, **실제로는 비교 방식(탭 횟수/구간 개수) 수정이 맞는 해결**이다.

---

## 3. 난이도 다시 올리기 (옵션)

- **의미**: “같은 리듬”이면 통과시키는 건 유지하되, **리듬이 얼마나 정확해야 같은 키로 인정할지**를 다시 조금 더 엄격하게.
- **방법**: `RHYTHM_QUANTIZE_STEP`을 **0.2 → 0.15**로 되돌리기.
  - **0.15**: 허용 범위 약 ±7.5% (기존 설계).
  - **0.2**: 허용 범위 약 ±10%.
- **호환성**:
  - **이미 0.2로 가입한 사용자**: 로그인 시 서버에서 step 0.2와 0.15 **둘 다** 시도하므로 그대로 로그인 가능.
  - **신규 가입자**: 0.15로 해시 저장 → 조금 더 정확한 리듬만 통과 (보안·일관성 상승).

---

## 4. 적용 여부

- **난이도 유지(0.2)**: 지금 상태 유지. 탭 횟수 수정만으로도 3번 실패는 해소됨.
- **난이도 상향(0.15)**: 클라이언트·서버의 **신규 가입/판정**만 0.15로 하고, 로그인 검증은 0.15 + 0.2 둘 다 시도 → 위 호환성 유지.

**난이도 상향(0.15 복원)** 적용함. 신규는 0.15, 로그인 시 0.15·0.2 둘 다 검사해 기존 0.2 가입자 호환.
