# 아이디 + 리듬 패턴 로그인 — 구체 설계안

비밀번호보다 **쉽고 빠른** 로그인: 닉네임 입력 후 **로고(또는 버튼)를 정해진 박자로만 탭**하면 로그인.

---

## 요약 (한눈에 보는 판)

| 항목 | 내용 |
|------|------|
| **로그인** | 닉네임 1칸 → 로고 탭(리듬) → **자동 전송** (별도 로그인 버튼 없음) |
| **가입** | 닉네임 → 리듬 2회 입력(일치 시 등록) → (선택) 추천인 |
| **저장** | 구간 비율을 **0.1 단위 양자화**한 문자열 → bcrypt 해시 → `rhythmHash` |
| **검증** | 입력도 같은 양자화 → 해시 비교 (오차는 양자화로 흡수) |
| **최소** | 탭 **4회 이상** (구간 3개 이상), 시도 제한(예: 5회/분) |

---

## 1. 사용자 흐름 (비밀번호보다 빠르게)

### 1.1 로그인 (2단계만)

| 단계 | 동작 | 비고 |
|------|------|------|
| 1 | **닉네임 입력** (한 칸) | 기존과 동일 |
| 2 | **「리듬 입력」 영역(로고)을 설정한 박자대로 탭** | 탭 끝나면 자동 전송, 버튼 한 번 더 누를 필요 없음 |

- 비밀번호 칸 없음 → **타이핑 한 번 덜함**, **Caps Lock/자동완성 신경 안 써도 됨.**
- 제출 버튼 없이 **리듬 입력이 끝나면 자동 로그인 요청** 가능 → 클릭 한 번 절약.

### 1.2 회원가입 (리듬 설정)

| 단계 | 동작 |
|------|------|
| 1 | 닉네임 입력 |
| 2 | **리듬 등록**: 로고를 원하는 박자로 2번 입력 (1차 입력 → 확인용 재입력). 두 번이 비슷하면 등록 성공. |
| 3 | (선택) 추천인 |

- 비밀번호·비밀번호 확인 대신 **「리듬 한 번 → 리듬 다시 한 번」** 으로 대체.

---

## 2. 데이터 설계

### 2.1 리듬의 표현

- **한 번의 리듬 입력** = 연속된 탭 사이의 **구간 길이** 배열.  
  예: 탭 시각 `[0, 400, 900, 1100]` (ms) → 구간 `[400, 500, 200]` (ms).
- **저장하는 값**: 절대 시간이 아니라 **상대 비율**만 사용 (기기/지연 차이 보정).
  - 구간 배열을 **합이 1이 되도록 정규화**  
    예: `[400, 500, 200]` → 합 1100 → `[400/1100, 500/1100, 200/1100]` ≈ `[0.364, 0.455, 0.182]`.
- **저장 형태(서버)**  
  - **옵션 A(권장)**: 정규화된 비율 배열을 **고정 소수 자리로 문자열화** 후 bcrypt 해시.  
    예: `"0.36,0.45,0.18"` → `rhythmHash`.  
  - **옵션 B**: 비율 배열을 그대로 JSON 등으로 저장하고, 검증 시만 비교(유출 시 재현 가능하므로 2FA용이나 저위험에만 권장).

### 2.2 사용자 스키마 추가

- 기존: `passwordHash`.
- 리듬 전용 시:
  - `rhythmHash` (string): bcrypt( 정규화 비율 문자열 ).
  - `passwordHash`: 제거하거나, **복구용/관리자용**으로만 유지(선택).

---

## 3. 검증 알고리즘 (오차 허용)

- **입력**: 사용자가 탭한 시각 배열 → 구간 길이 → 정규화 비율 배열 `u`.
- **저장값**: 등록 시 저장한 정규화 비율 배열 `s` (해시만 있으면 비교 시 비율을 직접 쓸 수 없으므로, **비율을 해시할 때 salt+round만 쓰고 검증은 “입력 비율 → 해시한 값이 저장 해시와 같은지”**로 하거나, **비율을 복호화 불가하게 저장하지 않고 “비교만 상수 시간”으로 하려면** 해시 대신 HMAC 등으로 검증 토큰만 저장하는 방식도 가능.  
  **실무적으로는** “비율 배열을 소수점 2자리로 반올림한 문자열”을 bcrypt 해시해 두고, **입력도 같은 방식으로 반올림한 뒤** bcrypt.compare로 비교하면 됨. 단, **오차**가 있으므로 입력 비율을 그대로 문자열화하면 매칭 실패 → **검증 시에는 “저장된 비율”을 복호화할 수 없으니, 서버에 원본 비율을 저장하지 않고 해시만 저장할 경우** 검증 절차가 달라짐.

**실제 구현 시 나눌 수 있는 두 가지:**

1. **해시만 저장 (원본 비율 미보관)**  
   - 등록: 비율 문자열 → bcrypt → `rhythmHash` 저장.  
   - 로그인: 입력 비율 문자열 → bcrypt.compare(입력해시, rhythmHash).  
   - 문제: 입력에 약간의 오차만 있어도 해시가 달라져서 **매칭 불가**.  
   - 따라서 **“오차 보정 후 하나의 대표 문자열”을 만드는 쪽**이 필요. 예: 비율을 **소수점 1자리로 반올림**한 문자열을 “대표값”으로 쓰고, 그 대표값을 bcrypt 해시해 저장. 로그인 시에는 입력 비율을 같은 방식으로 반올림해 대표값을 만들고, **그 대표값이 저장된 해시와 일치하는지** 확인.  
   - 더 나은 방법: **비율 배열을 “양자화”**해서 적은 개수의 구간만 쓰게 함. 예: 0.1 단위로만 (0.1, 0.2, … 1.0). 그러면 입력 비율을 가장 가까운 0.1 단위로 반올림한 뒤 문자열로 만들어 해시하고, 로그인 시에도 같은 양자화 규칙 적용 후 해시 비교.

2. **원본 비율 저장 (저위험 가정)**  
   - 등록: 정규화 비율 배열을 JSON 등으로 저장 (암호화 권장).  
   - 로그인: 입력 비율과 저장 비율을 **구간별로 오차 ±15% 이내인지** 비교. 전 구간 통과하면 성공.

**권장 (쉽고 빠르게 구현하면서 보안 유지):**

- **양자화 + 해시**  
  - 구간 비율을 **0.1 단위로 양자화** (예: 0.36 → 0.4, 0.455 → 0.45).  
  - 양자화된 비율 배열을 문자열로 (예: `"0.4,0.45,0.2"`) 만들어 **bcrypt 해시만** 저장 (`rhythmHash`).  
  - 로그인: 클라이언트가 탭 시각 전송 → 서버가 구간·정규화·양자화 → 같은 문자열 생성 → `bcrypt.compare(문자열, rhythmHash)`.  
  - 오차는 **양자화가 흡수** (0.1 단위면 ±5% 내외). 서버에는 **원본 비율을 보관하지 않아도** 됨.

## 4. UI/UX (쉽고 빠르게)

### 4.1 로그인 화면

- **상단**: 닉네임 입력 한 칸.
- **그 아래**:  
  - 문구 예: “아래 로고를 **당신만의 리듬**으로 탭해 주세요.”  
  - **탭 영역**: TornFi 로고(또는 큰 원형 버튼) 하나.  
  - 클릭/터치/스페이스바 중 하나 또는 모두 지원.
- **동작**:  
  - 닉네임 입력 후 포커스를 탭 영역으로 옮기거나, 탭 영역을 누르면 “리듬 입력 시작”.  
  - **첫 탭**부터 시각 기록, **N번째 탭 후 1.5초(또는 2초) 무입력**이면 “리듬 입력 완료”로 간주하고 **자동으로 로그인 API 호출**.  
  - 로딩 중에는 탭 영역 비활성화 + “확인 중…” 표시.
- **에러**: “리듬이 일치하지 않습니다. 다시 탭해 주세요.” 등. 실패 시 **다시 탭**만 하면 됨 (닉네임 다시 칠 필요 없음).

### 4.2 회원가입 화면

- 닉네임 → **리듬 2회 입력** (안내: “나만의 리듬을 두 번 입력해 주세요.”).  
  - 1차: 탭 → 1.5초 무입력 시 “첫 번째 입력 완료”.  
  - 2차: 같은 방식으로 탭.  
  - 두 입력의 **양자화된 비율 문자열이 동일**하면 등록 성공; 다르면 “두 번 입력이 달라요. 다시 입력해 주세요.” 후 처음부터.

### 4.3 리듬 규칙 (보안 + UX)

- **최소 탭 수**: 4회 이상 (구간 3개 이상). 너무 짧으면 조합 수 부족.
- **최대 구간 수**: 8~10개 정도 제한 (너무 길면 입력이 지연되고 재현도 어려움).
- **첫 구간 최소 길이**: 200ms 미만은 “실수 탭”으로 간주해 무시하거나, 최소 2~3구간 이상이어야 “완료”로 인정.

## 5. API 스펙 (요약)

### 5.1 회원가입 (리듬 사용)

- `POST /api/register`  
- Body: `{ displayName, rhythm, referrer? }`  
  - `rhythm`: 구간 길이 배열 (ms) 또는 **클라이언트에서 정규화+양자화한 비율 문자열**.  
  - 서버는 받은 rhythm을 양자화 규칙에 맞게 검증한 뒤 해시해 `rhythmHash` 저장.  
  - (선택) 기존 `password` 필드는 제거하거나, 복구용으로만 둠.

### 5.2 로그인 (리듬 사용)

- `POST /api/login`  
- Body: `{ id, rhythm }`  
  - `id`: 닉네임.  
  - `rhythm`: 구간 길이 배열(ms) 또는 클라이언트에서 정규화+양자화한 비율 문자열.  
- 서버: 해당 유저의 `rhythmHash`와 비교 (양자화 후 해시 비교 또는, 원본 비율 저장 시 오차 비교).  
- 성공 시: 기존과 동일하게 세션 생성·쿠키·`user` 반환.

### 5.3 기존 비밀번호와 공존 (선택)

- 로그인 시 `password`가 있으면 비밀번호 로그인, `rhythm`이 있으면 리듬 로그인.  
- 또는 “리듬 전용 계정”만 두고, 기존 계정은 비밀번호 유지 후 점진 이전.

## 6. 보안 체크리스트

- **시도 제한**: IP·닉네임당 분당 5~10회 실패 시 일시 잠금 또는 캡차.
- **최소 구간 수**: 3 이상 (탭 4회 이상).
- **양자화**: 0.1 단위 등으로 고정해 재현성 확보 + 해시 저장 가능.
- **HTTPS**: 필수.
- **세션/쿠키**: 기존과 동일 (변경 없음).

---

## 7. 구현 순서 제안

1. **공통 리듬 유틸** (클라이언트 + 서버 또는 서버만)  
   - 시각 배열 → 구간(ms) → 정규화 비율 → 양자화(0.1) → 문자열.
2. **서버**  
   - `rhythmHash` 저장/검증 로직.  
   - `POST /api/register`에 rhythm 처리 추가 (또는 비밀번호 제거 후 rhythm만).  
   - `POST /api/login`에 rhythm 분기 추가.
3. **클라이언트**  
   - 로그인 페이지: 닉네임 + 탭 영역, 탭 시각 수집 → 정규화·양자화 → `/api/login`에 `rhythm` 전송.  
   - 회원가입 페이지: 리듬 2회 입력 → 일치 시 `/api/register`에 `rhythm` 전송.
4. **마무리**  
   - 실패 메시지, 자동 전송 타이밍(1.5초 무입력), 최소 탭 수 검사.

이 순서대로 하면 **아이디 + 리듬만으로**, 비밀번호보다 **입력 단계가 적고**, **탭만으로 빠르게** 로그인할 수 있다.
